# Hand-off

다른(새) Claude 세션으로 작업을 넘기기 위한 스킬. 새 세션은 이 세션의 대화 맥락을 갖고 있지 않으므로,
**남겨진 기록만으로 작업을 그대로 이어갈 수 있어야 한다.**

새 Claude 세션이 시작 시 자동으로 읽는 건 `CLAUDE.md`와 메모리(`MEMORY.md` + 관련 메모리)뿐이다.
그래서 이 스킬은 **메모리에 `active-handoff` 재개 포인터 한 파일을 남기고, 그 본문에 인계 내용을 직접 담는다.**
대화 전체를 전사하지 않으므로 **빠르게 저장하고 세션을 닫을 수 있다.**

핵심은 "요약"이 아니라 "인계"다. 다음 세션이 코드/이력을 처음부터 다시 파악하지 않도록,
*어디까지 했고 / 지금 무엇이 열려 있고 / 다음에 무엇을 / 무엇을 조심해야 하는지*를 사실 기반으로 정확히 남긴다.

## 워크플로우

### Step 1 — 사실 확인
추측하지 말고 `git status`, `git log --oneline`, `git diff --name-status` 등으로 **사실을 확인한 뒤** 아래 §핸드오프 필수 항목을 모은다.
(대화 로그는 저장하지 않는다 — `save-conversation` 호출 없음.)

### Step 2 — `active-handoff` 메모리 작성
현재 프로젝트의 메모리 디렉토리에 **`active-handoff.md` 한 파일을 작성**한다(매번 같은 slug로 덮어써 인계 포인터가 누적되지 않게 한다). 이 파일의 **본문 자체가 인계 내용**이다 — 아래 §핸드오프 필수 항목을 빠짐없이 담고, 마지막에 §인계 완료 시 정리의 삭제 지시를 함께 써넣는다.

```markdown
---
name: active-handoff
description: 진행 중인 인계 작업 — 새 세션이 이어받을 재개 포인터
metadata:
  type: project
---
진행 중: <한 줄 주제> (<브랜치>)
- ✅ 완료: <커밋 해시/배포·DB 적용 등 검증된 사실>
- 🚧 진행 중: <정확히 어디까지>
- 📋 남은 작업: 1) … 2) … (순서대로, 실행 가능 수준)
- ⚠️ 주의: <순서 의존/되돌리기 어려움/함정>
- 📂 관련: plan=<경로>, 미커밋/untracked=<파일들>

**인계 완료(인계받은 작업을 다 끝냈거나 더 이상 이어갈 필요가 없을 때) 시 이 메모리(active-handoff.md)와 MEMORY.md의 해당 줄을 삭제**한다 — 남겨두면 끝난 작업을 진행 중으로 오인하게 하는 stale 포인터가 된다.
```

그리고 `MEMORY.md` 인덱스에 한 줄을 추가/갱신한다(예: `- [Active hand-off](active-handoff.md) — 진행 중: <주제>, 재개 포인터`). 이미 `active-handoff` 항목이 있으면 옛 내용을 지우고 이번 것으로 덮어쓴다.

### 마지막 — 결과 한 줄만 확인
저장이 끝나면 **결과 한 줄만** 알린다(예: `인계 메모리를 저장했습니다: memory/active-handoff.md`). 핸드오프 요약을 채팅에 다시 펼쳐 보여주지 않는다 — 내용은 이미 메모리 본문에 있고, 목적은 빠르게 세션을 닫는 것이다.

## 핸드오프 필수 항목

새 세션은 이 목록만 보고 움직이므로, 빠지면 그 부분은 새 세션이 처음부터 다시 파헤쳐야 한다. (이 내용이 `active-handoff.md` 본문에 들어간다.)

- **브랜치 / 작업 주제**: 현재 브랜치와 무엇을 하는 작업인지 한 줄.
- **완료된 것**: 끝나서 되돌릴 일 없는 것. 커밋 해시·배포·DB 적용 등 **검증된 사실**로(예: `커밋 3eb11ab`, `sandbox 배포·검증 완료`).
- **진행 중 / 미완**: 지금 열려 있는 작업이 **정확히 어디까지** 됐는지. "절반쯤"이 아니라 "X·Y 파일은 수정 완료, Z는 미반영"처럼 구체적으로.
- **남은 작업 (다음 세션 TODO)**: 새 세션이 이어서 할 일을 **순서대로**. 파일/메서드/시그니처 등 실행 가능한 수준으로.
- **위험·주의**: 하면 안 되는 것, 순서 의존(배포 순서 등), 되돌리기 어려운 작업, 알려진 함정. **인계에서 가장 값진 항목** — 빠지면 새 세션이 사고를 낼 수 있다.
- **관련 위치**: 계획서(plan) 경로, 미커밋/untracked 파일 목록, 참고 문서/이슈 링크, 핵심 파일 경로.

## 작성 원칙

- **사실 > 기억**: 커밋·배포·DB 상태는 명령으로 확인하고 적는다. "아마 됐을 것"은 적지 않는다.
- **다음 세션의 시점으로**: 이 대화를 못 본 사람이 읽는다고 가정. "아까 그거", "위에서 본" 같은 지시어 금지 — 매번 대상을 명시.
- **미완 작업의 재개점을 분명히**: 마지막으로 시도하다 실패/중단한 지점(예: "Edit이 충돌로 실패 → 재시도 필요")까지 적어 새 세션이 같은 함정을 반복하지 않게 한다.
- **간결하게**: 표·불릿 위주. 장황한 서사 대신 새 세션이 스캔하고 바로 행동할 수 있는 형태로.

## 인계 완료 시 정리

`active-handoff` 포인터는 **인계 완료(인계받은 작업을 다 끝냈거나 더 이상 이어갈 필요가 없을 때)** 시 수명을 다한다. 그 시점에 `active-handoff.md`와 `MEMORY.md`의 해당 줄을 **삭제**한다. 남겨두면 끝난 작업을 진행 중으로 오인하게 하는 stale 포인터가 된다.

이 삭제는 보통 **다음 세션**이 인계받은 작업을 마무리한 직후에 수행한다(메모리는 세션 시작 시 자동 로드되므로 포인터가 그 세션 컨텍스트에 떠 있다). 그래서 이 스킬은 위 §Step 2에서 같은 삭제 지시를 `active-handoff.md` 본문 끝에도 반드시 써넣어, 다음 세션이 그 지시를 직접 보고 정리하게 한다.

## 예시 (`active-handoff.md` 작성 형태)

```markdown
---
name: active-handoff
description: 진행 중인 인계 작업 — 새 세션이 이어받을 재개 포인터
metadata:
  type: project
---
진행 중: 결제 환불 배치 동시성 가드 (feature/REFUND-12)
- ✅ 완료: CAS 락 도입(커밋 a1b2c3d), sandbox 배포·1h 검증(중복 0)
- 🚧 진행 중: 복구 스케줄러에 락 필터 적용 — RecoveryService.java 수정 완료, 테스트 미작성
- 📋 남은 작업: 1) RecoveryServiceTest에 락 매처 추가 2) ./gradlew test 3) beta 배포
- ⚠️ 주의: beta/real 공유 DB → 락 키에 profile 포함 필수. real 먼저 배포 금지
- 📂 관련: plan=~/.claude/plans/refund-lock.md, 미커밋=RecoveryService.java

**인계 완료(인계받은 작업을 다 끝냈거나 더 이상 이어갈 필요가 없을 때) 시 이 메모리(active-handoff.md)와 MEMORY.md의 해당 줄을 삭제**한다 — 남으면 stale 포인터.
```
