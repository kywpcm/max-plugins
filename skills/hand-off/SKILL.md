# Hand-off

다른(새) Claude 세션으로 작업을 넘기기 위한 스킬. 새 세션은 이 세션의 대화 맥락을 갖고 있지 않으므로,
**남겨진 기록만으로 작업을 그대로 이어갈 수 있어야 한다.** 그래서 이 스킬은 두 가지를 한다:

1. `save-conversation` 스킬로 이번 세션 대화 로그를 저장한다(중복 작업 방지).
2. **메모리에 재개 포인터를 남겨**, 새 세션이 그 로그를 자동으로 집어들게 한다.
3. 핸드오프 요약을 채팅에도 보여줘 사용자가 결과를 확인하게 한다.

핵심은 "요약"이 아니라 "인계"다. 다음 세션이 코드/이력을 처음부터 다시 파악하지 않도록, *어디까지 했고 / 지금 무엇이 열려 있고 / 다음에 무엇을 / 무엇을 조심해야 하는지*를 사실 기반으로 정확히 남긴다.

## 워크플로우

### Step 1 — save-conversation 실행
`save-conversation` 스킬을 먼저 호출해 이번 세션 대화 로그를 `conv-logs/`에 저장한다. (디렉토리·파일명·증분 저장 규칙은 그 스킬을 따른다 — 여기서 중복 기술하지 않는다.)

단, 핸드오프 맥락이므로 그 로그의 **마지막 섹션은 반드시 `## 다음 세션 이어가기`** 로 구성한다. 일반 save-conversation의 토픽 요약 + 변경 파일에 더해, 아래 §핸드오프 필수 항목을 빠짐없이 담는다. 이 섹션이 인계의 본체다.

### Step 2 — 메모리에 재개 포인터 저장 (새 세션 자동 노출용)
새 Claude 세션은 `conv-logs/`를 스스로 찾아 읽지 않는다. 세션 시작 시 컨텍스트에 자동 로드되는 건 `CLAUDE.md`와 메모리(`MEMORY.md`)뿐이다. 그래서 **메모리에 재개 포인터를 남겨, 새 세션이 시작하자마자 "진행 중인 인계가 있고 어디를 읽으면 되는지"를 인지**하게 만든다 — 이게 로그와 새 세션을 잇는 다리다.

현재 프로젝트의 메모리 디렉토리에 **`active-handoff` 한 파일을 작성**한다(매번 같은 slug로 덮어써 인계 포인터가 누적되지 않게 한다):

```markdown
---
name: active-handoff
description: 진행 중인 인계 작업 — 새 세션이 이어받을 재개 포인터
metadata:
  type: project
---
진행 중: <한 줄 주제>. 재개하려면 `conv-logs/<이번에 저장한 로그 경로>`의 `## 다음 세션 이어가기` 섹션을 읽고 그 남은 작업부터 진행한다.

**이 인계 작업을 끝내면 이 메모리를 삭제**한다 — 완료 후에도 남아 있으면 다음 세션을 끝난 작업으로 잘못 이끄는 stale 포인터가 된다.
```

그리고 `MEMORY.md` 인덱스에 한 줄을 추가/갱신한다(예: `- [Active hand-off](active-handoff.md) — 진행 중: <주제>, 재개 포인터`). 이미 `active-handoff` 항목이 있으면 옛 내용을 지우고 이번 것으로 덮어쓴다.

### Step 3 — 핸드오프 요약을 채팅에 제시
저장·메모리 기록이 끝나면, 핸드오프 내용을 **채팅에도 간결한 표/불릿으로** 한 번 더 보여준다(저장한 로그 경로 포함). 사용자가 인계가 제대로 정리됐는지 바로 확인하게 하기 위함이다.

## 핸드오프 필수 항목

추측하지 말고 `git status`, `git log --oneline`, `git diff --name-status` 등으로 **사실을 확인한 뒤** 적는다.
새 세션은 이 목록만 보고 움직이므로, 빠지면 그 부분은 새 세션이 처음부터 다시 파헤쳐야 한다.

- **브랜치 / 작업 주제**: 현재 브랜치와 무엇을 하는 작업인지 한 줄.
- **완료된 것**: 끝나서 되돌릴 일 없는 것. 커밋 해시·배포·DB 적용 등 **검증된 사실**로(예: `커밋 3eb11ab`, `sandbox 배포·검증 완료`).
- **진행 중 / 미완**: 지금 열려 있는 작업이 **정확히 어디까지** 됐는지. "절반쯤"이 아니라 "X·Y 파일은 수정 완료, Z는 미반영"처럼 구체적으로.
- **남은 작업 (다음 세션 TODO)**: 새 세션이 이어서 할 일을 **순서대로**. 파일/메서드/시그니처 등 실행 가능한 수준으로.
- **위험·주의**: 하면 안 되는 것, 순서 의존(배포 순서 등), 되돌리기 어려운 작업, 알려진 함정. **인계에서 가장 값진 항목** — 빠지면 새 세션이 사고를 낼 수 있다.
- **관련 위치**: 계획서(plan) 경로, 미커밋/untracked 파일 목록, 참고 문서/이슈 링크, 핵심 파일 경로.

## 작성 원칙

- **사실 > 기억**: 커밋·배포·DB 상태는 명령으로 확인하고 적는다. "아마 됐을 것"은 적지 않는다.
- **다음 세션의 시점으로**: 이 대화를 못 본 사람이 읽는다고 가정. "아까 그거", "위에서 본" 같은 지시어 금지 — 매번 대상을 명시.
- **미완 작업의 재개점을 분명히**: 마지막으로 시도하다 실패/중단한 지점(예: "Edit이 충돌로 실패 → 재시도 필요")까지 적어 새 세션이 같은 함정을 반복하지 않게 한다.
- **간결하게**: 표·불릿 위주. 장황한 서사 대신 새 세션이 스캔하고 바로 행동할 수 있는 형태로.

## 예시 (채팅 제시 형태)

```
대화 내용을 저장했습니다: conv-logs/YYYYMM/DD/conv-<timestamp>.md

### 다음 세션 핸드오프
- 브랜치: feature/XXX — <작업 주제>
- ✅ 완료: <커밋 해시>, <배포/DB 적용 등 검증된 사실>
- 🚧 진행 중: <어디까지 됐는지 구체적으로>
- 📋 남은 작업: 1) … 2) … 3) … (순서대로, 실행 가능 수준)
- ⚠️ 주의: <배포 순서/되돌리기 어려운 것/함정>
- 📂 관련: plan=<경로>, 미커밋/untracked=<파일들>
```
