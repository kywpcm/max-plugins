# Comment Code

코드 파일에 한국어 주석을 자동으로 추가하는 스킬이다. 코드의 의도와 맥락을 명확히 전달하되, 과잉 주석을 방지하고 프로젝트 컨벤션을 존중한다.

## 사용법

### 파일 경로 인수가 있는 경우

```
/comment-code client/src/events/2025/example/index.jsx
```

- 지정된 파일을 읽고 주석을 추가한다.

### 파일 경로 인수가 없는 경우

```
/comment-code
```

- 대화형 모드로 진입한다.
- 사용자에게 주석 추가 범위를 물어본다:
  - **단일 파일**: 파일 경로를 입력받는다
  - **디렉토리**: 디렉토리 경로를 입력받고, 해당 디렉토리 내 모든 소스 파일에 주석을 추가한다
  - **변경된 파일**: `git diff`로 변경된 파일 목록을 가져와 주석을 추가한다

## 워크플로우

### 1단계: 파일 분석

대상 파일을 Read 도구로 읽고 다음을 파악한다:

- **언어 감지**: 파일 확장자로 언어를 판별한다 (`.js`, `.ts`, `.jsx`, `.tsx`, `.java`, `.py`, `.html`, `.css`, `.scss`, `.yaml`, `.yml`)
- **파일 유형 식별**: 구현 파일, 테스트 파일, 설정 파일, 스타일 파일, 데이터 파일 중 어느 것인지 판별한다
- **기존 주석 패턴 분석**: 파일에 이미 존재하는 주석의 스타일(언어, 형식, 위치)을 파악한다
- **코드 구조 파악**: 함수, 클래스, 컴포넌트, import/export, 조건문 등의 구조를 파악한다

### 2단계: 주석 전략 수립

파일 분석 결과를 바탕으로 주석 전략을 수립한다:

- `references/comment-styles.md`를 참조하여 언어별 주석 문법을 확인한다
- 4가지 주석 유형(Doc, 인라인, 섹션 구분, TODO/FIXME) 중 적용할 유형을 결정한다
- 기존 주석과 중복되지 않도록 계획한다
- 과잉 주석을 방지하기 위해 자명한 코드는 건너뛴다

### 3단계: 주석 삽입

Edit 도구를 사용하여 파일에 주석을 추가한다:

- 한 번에 하나의 주석 블록씩 추가한다
- 각 Edit 후 파일 구조가 깨지지 않았는지 확인한다
- 모든 주석은 **한국어**로 작성한다

### 4단계: 검증

주석 추가 후 코드가 정상 동작하는지 확인한다:

- 해당 언어의 린트 도구를 실행한다 (예: `npm run lint`, `eslint`)
- 관련 테스트가 있으면 실행한다
- 문법 오류가 발생하면 즉시 수정한다

## 주석 유형별 적용 규칙

### 1. Doc 주석

함수, 클래스, 컴포넌트의 상단에 추가하는 문서화 주석이다.

**적용 대상:**
- `export` 또는 `public` 접근 제한자가 있는 함수/메서드
- 파라미터가 3개 이상인 함수
- 복잡한 반환 타입을 가진 함수 (객체, 배열, Promise 등)
- React 컴포넌트 (함수형, 클래스형 모두)
- 클래스 선언

**적용하지 않는 대상:**
- getter/setter 같은 단순 접근자
- 이름만으로 용도가 명확한 함수 (예: `formatDate`, `isValid`)
- 파라미터가 0~1개이고 반환 타입이 단순한 함수
- 화살표 함수로 작성된 짧은 유틸리티 (한 줄 함수)

### 2. 인라인 주석

코드 라인 위 또는 옆에 추가하는 짧은 설명 주석이다.

**적용 대상:**
- 비즈니스 규칙이 담긴 조건문 (예: `if (age >= 19)` → `// 성인 인증 기준`)
- 복잡한 조건식 (3개 이상의 조건 조합)
- 정규식 패턴
- 매직 넘버 (예: `timeout: 3000` → `// 3초 타임아웃`)
- 외부 API 호출 시 특이사항
- 비직관적인 타입 캐스팅이나 변환

**핵심 원칙: "무엇을"이 아니라 "왜"를 설명한다.**

```javascript
// BAD: 배열을 필터링한다 (무엇을 - 코드를 그대로 번역)
const filtered = items.filter(item => item.active);

// GOOD: 비활성 상품은 목록에서 제외한다 (왜 - 비즈니스 의도)
const filtered = items.filter(item => item.active);
```

### 3. 섹션 구분 주석

파일 내 논리적 영역을 구분하는 주석이다.

**적용 대상:**
- 100줄 이상인 파일에서 논리적으로 구분되는 영역
- React 컴포넌트 내 state, effect, handler, render 영역
- 클래스 내 필드, 생성자, public 메서드, private 메서드 영역

**패턴:**
```javascript
// === State 관리 ===
const [count, setCount] = useState(0);
const [isOpen, setIsOpen] = useState(false);

// === 이벤트 핸들러 ===
const handleClick = () => { ... };

// === 렌더링 ===
return ( ... );
```

**적용하지 않는 대상:**
- 100줄 미만의 짧은 파일
- 이미 구조가 명확한 파일

### 4. TODO/FIXME 주석

미완성 기능, 잠재 버그, 임시 해결책을 표시하는 주석이다.

**형식:**
```javascript
// TODO: 페이지네이션 API 연동 후 무한스크롤로 교체
// FIXME: iOS Safari에서 스크롤 위치 복원 안 되는 이슈
// HACK: API 응답 지연으로 인한 임시 딜레이 (서버 개선 후 제거)
```

**적용 대상:**
- 하드코딩된 값이 추후 동적으로 변경될 예정인 경우
- 알려진 버그가 있지만 당장 수정하지 않는 경우
- 임시 해결책(workaround)을 적용한 경우
- 성능 최적화가 필요한 부분

## 특수 케이스 처리

### JSON 파일
- JSON은 주석을 지원하지 않는다.
- 주석이 필요한 경우, 같은 디렉토리에 `README.md`나 `.comments` 파일을 생성하여 설명을 작성하는 대안을 사용자에게 제안한다.
- JSONC(JSON with Comments) 형식이 지원되는 환경이면 `//` 주석을 사용할 수 있다.

### JSX/TSX 파일
- JSX 반환문 **외부**: `// 주석` (일반 JavaScript 주석)
- JSX 반환문 **내부**: `{/* 주석 */}` (JSX 주석 문법)
- JSX 내부에서 `//` 주석을 사용하면 문법 오류가 발생하므로 반드시 `{/* */}` 를 사용한다

### HTML 파일
- `<!-- 설명 -->` 형태만 사용한다
- `<script>` 태그 내부는 JavaScript 주석 문법(`//`, `/* */`)을 적용한다
- `<style>` 태그 내부는 CSS 주석 문법(`/* */`)을 적용한다

### 테스트 파일
- AAA(Arrange-Act-Assert) 패턴에 맞춰 영역 구분 주석을 추가한다:
```javascript
it('장바구니에 상품을 추가한다', () => {
  // Arrange: 테스트 데이터 준비
  const item = { id: 1, name: '테스트 상품' };

  // Act: 장바구니 추가 실행
  addToCart(item);

  // Assert: 결과 검증
  expect(cart.items).toContain(item);
});
```

### 설정 파일 (webpack, babel, eslint 등)
- 각 설정 항목의 **용도**를 설명하는 인라인 주석을 추가한다
- 비표준 설정이나 커스텀 설정에 우선적으로 주석을 추가한다

## 기존 주석 보존 전략

- **유지**: 기존 주석은 삭제하거나 수정하지 않는다
- **중복 방지**: 기존 주석과 동일한 내용의 주석을 추가하지 않는다
- **보완**: 기존 주석이 불완전하면(파라미터 설명 누락 등) 보완한다. 단, 기존 주석의 스타일과 언어를 따른다 (영어 주석이 있으면 영어로 보완)
- **스타일 일관성**: 파일 내 기존 주석 스타일(들여쓰기, 구분선 패턴 등)을 분석하여 동일하게 적용한다
